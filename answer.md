## 3) Реализация интеграционных механизмов

### 3.1 Интеграция с локальным сервисом Ollama (Aspect-Based Sentiment Analysis)

#### Цель интеграции

Автоматический анализ текстов отзывов потребителей с использованием локального языкового модели (LLM) для выполнения задачи Aspect-Based Sentiment Analysis (ABSA). Система должна извлекать из каждого отзыва:

- общую тональность (положительная, отрицательная, смешанная);
- категорийные аспекты товара (производительность, дизайн, батарея, звук, цена и т.д.);
- тональность каждого аспекта (положительная, отрицательная);
- подтверждающие цитаты из исходного текста отзыва.

#### Протокол и формат обмена

Интеграция реализована через **REST API**:

- **Базовый URL**: `http://localhost:11434` (конфигурируется через переменную окружения `OLLAMA_URL`)
- **Endpoint**: `/api/generate`
- **Метод HTTP**: POST
- **Формат запроса**: JSON
- **Формат ответа**: JSON

Структура запроса:

```json
{
  "model": "qwen3:4b-instruct",
  "prompt": "<текст промпта с инструкциями анализа>",
  "stream": false,
  "options": {
    "temperature": 0.05
  }
}
```

Структура ответа:

```json
{
  "response": "<JSON-объект с результатом анализа>"
}
```

#### Реализация клиента интеграции

Модуль `analyzer_simple.py` содержит функцию `analyze_review()`, которая реализует клиента для взаимодействия с Ollama:

1. **Формирование промпта**: Функция генерирует специализированный промпт, содержащий:
   - Описание роли анализатора (ROLE: эксперт по анализу текстов и ABSA)
   - Список разрешённых категорий аспектов (23 категории: сервис, дизайн, производительность, батарея, экран, звук, цена, комплектация, эргономика, безопасность, камера, вес, корпус и т.д.)
   - Строгие инструкции по формату вывода
   - Примеры входа и выхода

2. **Отправка запроса**: Используется библиотека `requests` с таймаутом 180 секунд

3. **Парсинг ответа**:
   - Проверка статус-кода HTTP
   - Очистка ответа от markdown-разметки и блоков `<think>`
   - Парсинг JSON-ответа
   - Валидация структуры и санитизация данных

#### Обработка ошибок

Функция `analyze_review()` реализует следующие механизмы обработки ошибок:

1. **Таймауты** (180 сек): Возвращает `("error", "Ollama timeout (>180 сек)", [])`
2. **Ошибки подключения**: `("error", "Ollama недоступен (Connection refused)", [])`
3. **Ошибки HTTP**: `("error", f"Ollama HTTP {resp.status_code}", [])`
4. **Ошибки парсинга JSON**: `("error", f"JSON parse error: {e}", [])`
5. **Прочие исключения**: Перехват и логирование всех ошибок

### 3.2 Интеграция с CSV-источниками данных (файловый обмен)

#### Цель интеграции

Массовая загрузка отзывов из внешних CSV-файлов в систему.

#### Протокол и формат обмена

- **Протокол**: Файловый обмен через HTTP multipart/form-data
- **Формат данных**: CSV (UTF-8 / cp1251)
- **Обязательные колонки**: `product_name`, `review_text`
- **Опциональные колонки**: `category`, `customer_name`, `created_at`

#### Обработка ошибок

1. **Ошибки кодировки**: Попытка UTF-8 с BOM → fallback на cp1251
2. **Валидация структуры**: Проверка наличия заголовков CSV
3. **Валидация данных**:
   - Проверка пустых обязательных полей
   - Проверка минимальной длины отзыва (5+ символов)
4. **Ошибки анализа**: Отзыв сохраняется со статусом "pending", ошибка записывается в список
5. **Возврат результата**: JSON с инфо о добавленных/проанализированных отзывах и списком ошибок (max 30)

## 4) Реализация вспомогательных механизмов

### 4.1 Аутентификация и авторизация

**Хранение пользователей**: Таблица `users` с хешированными паролями (werkzeug)

**Процесс входа**:

```python
user = db.query_one("SELECT * FROM users WHERE email = ?", (creds.email,))
if not user or not check_password_hash(user["password"], creds.password):
    return {"success": False, "message": "Неверный логин или пароль"}
return {"success": True, "access_token": f"user_{user['id']}", ...}
```

**Тестовый пользователь**: email="user", password="user"

### 4.2 Логирование

- **Консольное логирование** (stdout):
  - Инициализация БД
  - Запуск сервера
  - Прогресс загрузки отзывов
  - Ошибки анализа (в консоль и в массив `errors[]`)

- **Отсутствующие механизмы**: Логирование на диск, структурированное логирование, уровни

### 4.3 Резервное копирование

- **SQLite с WAL**: Включен режим WAL (Write-Ahead Logging) для надежности
- **Инструмент очистки**: `clean_db.py` для ручной очистки данных
- **Отсутствующие механизмы**: Автоматическое резервное копирование, шифрование, репликация

### 4.4 Регламентные задания

- **Инициализация при запуске**: `db.init_tables()` и `db.init_test_user()`
- **Отсутствующие механизмы**: Встроенные фоновые задачи, планировщик (рекомендуется APScheduler или Celery)

### 3.2 Оценка выполнения нефункциональных требований

#### **NFR-01: Производительность**

**Требование**: Система должна обрабатывать запросы пользователя с минимальной задержкой, выполнять анализ отзывов в приемлемое время.

**Реализация**:

- **GET /products**: 50-200 мс (зависит от количества товаров)
- **GET /reviews**: 100-500 мс (парсинг JSON аспектов)
- **GET /stats**: 30-100 мс (GROUP BY операция)
- **GET /report/html**: 200-800 мс (построение HTML с диаграммами)
- **POST /upload/csv**: 3-10 мин для 100 отзывов (зависит от скорости Ollama: 2-5 сек на отзыв)

**Оценка**: ✅ **Выполнено частично**. Для типовых операций (чтение данных) производительность приемлема. Для анализа отзывов время зависит от производительности Ollama. Рекомендуется оптимизация через индексирование БД и асинхронную обработку анализа.

---

#### **NFR-02: Безопасность**

**Требование**: Система должна защищать учетные данные пользователей, предотвращать несанкционированный доступ.

**Реализация**:

- **Хеширование паролей**: Пароли хранятся в виде хеша с использованием функции `generate_password_hash()` из werkzeug. При аутентификации выполняется проверка `check_password_hash()`.
- **SQL injection**: Все параметризованные запросы используют вопросительные знаки (placeholders) вместо конкатенации строк.
- **CORS**: Включена поддержка CORS для всех источников (allow_origins=["*"]), что необходимо для разработки, но не рекомендуется для production.
- **Access Control**: Текущая версия не реализует строгую проверку прав доступа; все endpoints доступны авторизованным пользователям.

**Оценка**: ⚠️ **Выполнено частично**. Реализована основная защита (хеширование паролей, защита от SQL injection). Отсутствуют: JWT-токены в заголовках Authorization, HTTPS, rate limiting, логирование попыток входа.

---

#### **NFR-03: Надежность и отказоустойчивость**

**Требование**: Система должна корректно обрабатывать сбои внешних сервисов (Ollama), предотвращать потерю данных.

**Реализация**:

- **Обработка таймаутов**: При превышении 180 сек отзыв сохраняется со статусом "pending", ошибка записывается в лог.
- **Обработка сбоев Ollama**: При Connection refused или HTTP ошибках система продолжает работу, возвращая информацию об ошибках пользователю.
- **WAL режим SQLite**: Включен режим журнала Write-Ahead Logging (PRAGMA journal_mode=WAL) для предотвращения повреждения БД при неправильном выключении.
- **Foreign keys**: Включены ограничения целостности (PRAGMA foreign_keys=ON).
- **Отсутствие автоматических повторных попыток**: При ошибке анализа отзыв сохраняется со статусом "pending", повторный анализ возможен через переиндексирование вручную.

**Оценка**: ✅ **Выполнено**. Система устойчива к сбоям внешних сервисов. Данные защищены благодаря WAL режиму. Рекомендуется добавить автоматические повторные попытки (retry logic) с экспоненциальной задержкой для анализа pending отзывов.

---

#### **NFR-04: Масштабируемость**

**Требование**: Система должна иметь возможность обработки больших объемов данных (1000+ отзывов).

**Реализация**:

- **Асинхронная загрузка CSV**: Используется async функция `async def upload_csv()`, позволяющая обрабатывать параллельные запросы.
- **Итеративная обработка**: CSV обрабатывается строка за строкой, а не целиком в памяти.
- **БД масштабирование**: SQLite подходит для ~100K записей, для большего объема рекомендуется PostgreSQL.
- **Отсутствие кэширования**: Каждый GET запрос выполняет новый SQL запрос. Для высоконагруженных систем рекомендуется добавить кэширование (Redis).

**Оценка**: ⚠️ **Выполнено частично**. Система может обработать CSV с 1000+ отзывов, но для миллионных объемов данных требуется оптимизация архитектуры (кэширование, индексирование, шардирование БД).

---

#### **NFR-05: Удобство использования**

**Требование**: Веб-интерфейс должен быть интуитивным и простым в использовании для маркетингового аналитика.

**Реализация**:

- **Трёхступенчатый интерфейс**: login.html → upload.html → dashboard.html (логичный поток).
- **Визуальные диаграммы**: Chart.js диаграмма (тональность), стacked bar charts (аспекты).
- **Таблицы с цитатами**: Отзывы с подтверждающими цитатами для быстрого анализа.
- **Адаптивный дизайн**: CSS использует flexbox и grid для адаптации под разные размеры экрана.
- **Экспорт в PDF**: Встроенная поддержка печати браузером с оптимизацией CSS для печати.
- **Отсутствие сложных процедур**: Загрузка CSV и просмотр отчетов не требуют дополнительной конфигурации.

**Оценка**: ✅ **Выполнено**. Интерфейс интуитивен и прост в использовании. Все основные функции доступны в несколько кликов. Рекомендуется добавить справку/туториал для новых пользователей.

---

#### **NFR-06: Совместимость**

**Требование**: Система должна работать на различных операционных системах и браузерах.

**Реализация**:

- **Backend**: Python 3.9+ (кроссплатформенный), FastAPI (независим от ОС).
- **БД**: SQLite (встроенная в Python, работает везде).
- **Фронтенд**: HTML5 + CSS3 + JavaScript (ES6), совместим с современными браузерами (Chrome, Firefox, Safari, Edge).
- **Кодировки**: Поддержка UTF-8 и cp1251 для CSV файлов.

**Оценка**: ✅ **Выполнено**. Система кроссплатформенна и совместима с современными браузерами.

---

#### **NFR-07: Логирование и мониторинг**

**Требование**: Система должна логировать события и ошибки для последующего анализа.

**Реализация**:

- **Консольное логирование**: Ошибки, прогресс загрузки, инициализация выводятся в stdout.
- **Накопление ошибок**: Ошибки при загрузке CSV накапливаются в массиве `errors[]` и возвращаются пользователю.
- **Отсутствие логирования на диск**: Логи не сохраняются в файлы, нет ротации логов.
- **Отсутствие метрик**: Нет сбора метрик производительности, нет интеграции с системами мониторинга (Prometheus, ELK).

**Оценка**: ⚠️ **Выполнено частично**. Базовое логирование в консоль работает, но для production требуется логирование на диск с ротацией и интеграция с системами мониторинга.

## Резюме архитектуры

### Ключевые компоненты системы

| Компонент       | Технология         | Назначение                                      |
| --------------- | ------------------ | ----------------------------------------------- |
| **Веб-сервер**  | FastAPI + Uvicorn  | REST API, веб-интерфейс                         |
| **База данных** | SQLite + WAL       | Хранилище отзывов, товаров, результатов анализа |
| **LLM сервис**  | Ollama (локальный) | Анализ отзывов (ABSA)                           |
| **Фронтенд**    | HTML + JavaScript  | Веб-интерфейс для пользователей                 |
| **Скрипты**     | Python CLI         | Загрузка CSV, очистка БД                        |

### Таблица скорости выполнения операций

| Операция                    | Время выполнения | Факторы влияния                           |
| --------------------------- | ---------------- | ----------------------------------------- |
| `/auth/login`               | 10-50 мс         | Хеширование пароля                        |
| GET `/products`             | 50-200 мс        | Количество товаров, JOIN операции         |
| GET `/reviews`              | 100-500 мс       | Количество отзывов, парсинг JSON аспектов |
| GET `/stats`                | 30-100 мс        | Количество отзывов, GROUP BY              |
| `/upload/csv` (100 отзывов) | 3-10 минут       | Скорость Ollama (2-5 сек на отзыв)        |
| `/report/html`              | 200-800 мс       | Количество отзывов, построение HTML       |
